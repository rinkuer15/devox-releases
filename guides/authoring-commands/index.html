Authoring Commands

This guide explains how to write effective commands for Devox‚Äôs AI workflow system. Commands are the building blocks of workflows - each command is a prompt template that instructs the AI agent what to do.

What is a Command?

Section titled ‚ÄúWhat is a Command?‚Äù

A command is a markdown file that serves as a detailed instruction set for an AI agent. When a workflow executes a step like - command: investigate-issue, Devox:

  1. Loads the command file from .devox/commands/investigate-issue.md
  2. Substitutes variables like $ARGUMENTS with actual values
  3. Sends the entire document as a prompt to the AI
  4. The AI follows the instructions and produces output

Commands are prompts, not code. They guide AI behavior through clear instructions.

File Format

Section titled ‚ÄúFile Format‚Äù

Commands live in .devox/commands/ relative to the working directory and are loaded at runtime.

CLI vs Server: The CLI reads commands from wherever you run it (sees uncommitted changes). The server reads from ~/.devox/workspaces/owner/repo/, which only syncs from the remote before worktree creation ‚Äî so changes must be committed and pushed for the server to pick them up.

The Golden Rule: Artifacts Are Everything

Section titled ‚ÄúThe Golden Rule: Artifacts Are Everything‚Äù

The artifact you produce IS the specification for the next step.

In multi-step workflows, agents don‚Äôt share memory. The ONLY way to pass information between steps is through artifacts - files saved to disk.

Why This Matters

Section titled ‚ÄúWhy This Matters‚Äù
  • No shared context: Each workflow node can run with context: fresh
  • Resumability: If a step fails, the artifact preserves progress
  • Auditability: Artifacts create a paper trail of AI decisions
  • Handoff quality: The artifact determines if the next step succeeds

Command Structure

Section titled ‚ÄúCommand Structure‚Äù

Phase-Based Organization

Section titled ‚ÄúPhase-Based Organization‚Äù

Break commands into clear phases. This helps the AI:

  • Know where it is in the process
  • Self-verify before proceeding
  • Recover if something fails

Why Phases Work

Section titled ‚ÄúWhy Phases Work‚Äù
  1. Chunked reasoning: AI handles complex tasks better in pieces
  2. Self-verification: Checkpoints force the AI to validate progress
  3. Debugging: When something fails, you know which phase
  4. Consistency: Similar structure across commands = predictable behavior

Common Phase Patterns

Section titled ‚ÄúCommon Phase Patterns‚Äù
Phase NamePurposeExample Actions
LOADGather inputs and contextRead files, fetch from GitHub, parse arguments
EXPLOREUnderstand the codebaseSearch for patterns, trace code flow
ANALYZEForm conclusionsRoot cause analysis, design decisions
GENERATEProduce outputWrite artifact, create files
VALIDATEVerify correctnessRun tests, check types, review output
COMMITSave to gitStage, commit, push
REPORTCommunicate resultsOutput summary to user

Variable Substitution

Section titled ‚ÄúVariable Substitution‚Äù

Devox replaces variables in command text before sending to the AI. The most commonly used variables in commands:

VariableValue
$ARGUMENTS / $USER_MESSAGEUser‚Äôs input message
$1, $2, $3Positional arguments (direct invocation only)
$ARTIFACTS_DIRPre-created artifacts directory for this workflow run
$BASE_BRANCHBase branch (auto-detected or configured)
$DOCS_DIRDocumentation directory path (default: docs/)
$WORKFLOW_IDUnique workflow run ID
$CONTEXTGitHub issue/PR context (if available)

See the Variable Reference for the complete list, including $LOOP_USER_INPUT, $REJECTION_REASON, node output references, substitution order, and context variable behavior.
