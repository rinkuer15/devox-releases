Per-Node MCP Servers

DAG workflow nodes support a mcp field that attaches MCP (Model Context Protocol) servers to individual nodes. Each node gets exactly the external tools it needs ‚Äî GitHub, Linear, Postgres, etc. ‚Äî without over-provisioning.

Claude only ‚Äî Codex nodes will warn and ignore the mcp field.

MCP config files are JSON objects where each key is a server name and the value is a server configuration. Three transport types are supported:

Environment Variable Expansion

Section titled ‚ÄúEnvironment Variable Expansion‚Äù

Values in env and headers fields support $VAR_NAME references that are expanded from process.env at execution time.

Rules:

  • Pattern: $UPPER_CASE_VAR (matches [A-Z_][A-Z0-9_]*)
  • Only env and headers values are expanded ‚Äî command, args, url are left untouched
  • Undefined vars are replaced with empty string and a warning is shown: Warning: Node 'X' MCP config references undefined env vars: VAR_NAME
  • Expansion happens at execution time, not when the workflow YAML is loaded

Why file-based? MCP configs often contain secrets (API tokens, database URLs). Workflow YAML files are committed to git. By keeping configs in separate JSON files, you can gitignore them or rely on env var references so secrets never appear in source.

Automatic Tool Wildcards

Section titled ‚ÄúAutomatic Tool Wildcards‚Äù

When a node loads MCP servers, tool wildcards are automatically added to allowedTools. For servers named github and postgres, the node gets:

  • mcp__github__*
  • mcp__postgres__*

This means all tools from those servers are immediately available without manually listing them. The wildcards merge with any existing allowed_tools on the node.

Limitations

Section titled ‚ÄúLimitations‚Äù
  • Claude only ‚Äî Codex nodes warn and ignore the mcp field. Configure MCP servers globally in the Codex CLI config instead.
  • Haiku model ‚Äî Tool search (lazy loading for many tools) is not supported on Haiku. You‚Äôll see a warning. Consider using Sonnet or Opus for MCP nodes.
  • No load-time validation ‚Äî The MCP config file is read at execution time, not when the workflow YAML is loaded. A typo in the path won‚Äôt surface until the node runs.
  • No inline config ‚Äî MCP configs must be in a separate JSON file, not inline in YAML. This is intentional ‚Äî it keeps secrets out of version-controlled workflow files.
