Resuming and Rerunning Workflows

When a workflow fails mid-run, Devox can skip nodes that already completed and continue from the failure point. This page explains all the resume and rerun options available, what each one does, and what limitations exist — especially for multi-phase workflows that call sub-workflows internally.

How resume works

Section titled “How resume works”

Devox persists every node’s completion status and output to the database during a run. When a workflow fails, the run is marked failed in the database along with which nodes completed.

On the next execution of the same workflow on the same working path, Devox automatically detects the prior failed run and replays it:

  1. Loads all previously-completed node outputs from the database
  2. Skips every completed node (marks them skipped_prior_success)
  3. Re-runs only from the first failed or not-yet-run node onward

This happens automatically — you do not need to tell Devox which nodes to skip.

Note: AI session context from prior nodes is not restored on resume. Nodes that need previous AI output should read it from artifact files rather than relying on the in-memory session chain.

Web UI — Dashboard

Section titled “Web UI — Dashboard”

The Dashboard provides three resume-related actions:

ActionWhere to find itWhat it does
Resume buttonRun card (active/failed runs)Marks the run as resumable — informational only; does not re-dispatch execution
Rerun buttonWorkflow history tableCreates a new conversation and dispatches /workflow run … — this is the action that actually re-runs with skip logic
Rerun All FailedDashboard bulk toolbarDispatches all failed/cancelled runs in one click

Important: The Resume button on a run card does not trigger execution. It validates the run is in a resumable state and shows a confirmation. To actually restart execution, use the Rerun button in the history table.

Step-by-step: resume a failed run from the Web UI

Section titled “Step-by-step: resume a failed run from the Web UI”
  1. Open the Dashboard
  2. Click the History tab to see completed/failed runs
  3. Find the failed run
  4. Click the Rerun button (↩ icon in the history row)
  5. Devox creates a new conversation, dispatches the workflow, and auto-skips all previously-completed nodes

Resume granularity

Section titled “Resume granularity”

Resume operates at the orchestrator (top-level) node level, not inside sub-workflows.

ScenarioSupportedDetail
Orchestrator fails at run-design-overview node → resume skips earlier sub-workflows✅ YesOrchestrator-level nodes run-concept-validation, gate-1, run-enterprise-context, gate-2 are all skipped
Resume resumes exactly from the failed sub-workflow✅ Yesrun-design-overview re-runs in full

Running a sub-workflow standalone with existing artifacts

Section titled “Running a sub-workflow standalone with existing artifacts”

Sub-workflows like concept-validation, design-overview, and build-ready-plan can be run standalone for development or targeted reruns. They detect whether they are being called by an orchestrator by checking for $ARTIFACTS_DIR/input/context.json:

  • Standalone (context.json absent): prompts for input, runs approval gates
  • Sub-workflow mode (context.json present): reads from manifest, skips gates

When running standalone, the sub-workflow always starts with a fresh empty $ARTIFACTS_DIR. To reuse artifacts from a prior orchestrator run, set the DEVOX_ARTIFACTS_OVERRIDE environment variable:

This is an advanced escape hatch — the typical path is to rerun the orchestrator and let it auto-skip completed phases.
