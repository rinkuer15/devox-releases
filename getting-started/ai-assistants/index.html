AI Assistants

Devox supports four AI provider backends out of the box. All share the same workflow DSL — only the setup and provider-specific options differ. You can mix providers in the same workflow: one node on Claude, the next on Codex.

ProviderIDBest forCost
Claude Code (Anthropic)claudeBest overall capability, extended thinking, MCP, skillsAnthropic API billing
Codex CLI (OpenAI)codexStrong reasoning, web search, structured outputOpenAI API billing
GitHub CopilotcopilotNo extra API cost if you have a Copilot subscriptionIncluded in Copilot sub
Pi (Community Provider)piMulti-model router — Gemini, Llama, Qwen, Mistral and more via one providerPay-as-you-go via chosen backend API key

Claude Code

Section titled “Claude Code”

Recommended for Claude Pro/Max subscribers.

Devox does not bundle Claude Code. Install it separately, then in compiled Devox binaries, point Devox at the executable. In dev (bun run), Devox finds it automatically via node_modules.

Section titled “Authentication Options”

  1. Global Auth (set to true): Uses credentials from claude /login
  2. Explicit Tokens (set to false): Uses tokens from env vars below
  3. Auto-Detect (not set): Uses tokens if present in env, otherwise global auth

GitHub Copilot

Section titled “GitHub Copilot”

Recommended for GitHub Copilot subscribers (Individual, Business, or Enterprise).

Devox uses the @github/copilot-sdk — no separate binary install required. You only need an active Copilot subscription and a valid credential.

Option 1 — Stored gh CLI credentials (recommended)

Section titled “Option 1 — Stored gh CLI credentials (recommended)”

If you have the GitHub CLI installed, this is the zero-token approach:

Option 2 — Explicit token (CI/CD, no gh CLI, or Windows fallback)

Section titled “Option 2 — Explicit token (CI/CD, no gh CLI, or Windows fallback)”

Set COPILOT_GITHUB_TOKEN to a GitHub OAuth user token (gho_) or GitHub App user token (ghu_) that has the Copilot scope:

Getting a token:

Token compatibility

GH_TOKEN / GITHUB_TOKEN are not used for Copilot — they are for the GitHub forge adapter (clone / PR / issues) only. Devox automatically strips them from the Copilot subprocess environment so they cannot interfere even if they lack the Copilot scope.

Classic Personal Access Tokens (ghp_ / github_pat_) are rejected by the Copilot API. Use gho_ or ghu_ tokens only.

Pi (Community Provider)

Section titled “Pi (Community Provider)”

One adapter, ~20 LLM backends. Pi (@mariozechner/pi-coding-agent) is a community-maintained coding-agent harness that Devox integrates as the first community provider. It unlocks Anthropic, OpenAI, Google (Gemini + Vertex), Groq, Mistral, Cerebras, xAI, OpenRouter, Hugging Face, and more under a single provider: pi entry.

Pi is registered as builtIn: false — it validates the community-provider seam rather than being a core-team-maintained option. If it proves stable and valuable it may be promoted to builtIn: true later.

Install

Section titled “Install”

Pi is included as a dependency of @devox/providers — no separate install needed. It’s available immediately.

Authenticate

Section titled “Authenticate”

Pi supports both OAuth subscriptions and API keys. Devox’s adapter reads your existing Pi credentials from ~/.pi/agent/auth.json (written by running pi → /login) AND from env vars — env vars take priority per-request so codebase-scoped overrides work.

OAuth subscriptions (run pi /login locally):

  • Anthropic Claude Pro/Max
  • OpenAI ChatGPT Plus/Pro
  • GitHub Copilot
  • Google Gemini CLI
  • Google Antigravity

API keys (env vars):

OptionValuesDescription

Tip

Pi supports effortControl and thinkingControl via the effort field. Use effort: max in your workflow YAML for maximum reasoning (Pi translates it to xhigh internally). The Claude-specific thinking: { type: enabled, budget_tokens: N } object form is NOT supported by Pi — use effort instead.

Choosing a provider

Section titled “Choosing a provider”
If you have…UseWhy
Anthropic API keyclaudeBest overall capability; extended thinking; skills; MCP; per-node tool control
OpenAI API keycodexStrong reasoning; built-in web search; structured JSON output
GitHub Copilot subscriptioncopilotNo extra API cost; shares your existing Copilot auth
Google / Groq / OpenRouter / other keypiWidest model choice; no extra binary; run Gemini, Llama, Qwen and more

Tip

You can mix providers in the same workflow. For example: a plan node runs on Claude for best reasoning, an implement node runs on Codex, and a review node runs on Copilot — each bringing its strengths to its step.

How Assistant Selection Works

Section titled “How Assistant Selection Works”
  • Assistant type is set per codebase via the assistant field in .devox/config.yaml or the DEFAULT_AI_ASSISTANT env var
  • Once a conversation starts, the assistant type is locked for that conversation
  • DEFAULT_AI_ASSISTANT (optional) is used only for new conversations without codebase context
  • Workflows can override the assistant on a per-node basis with provider and model fields
  • Configuration priority: workflow-level options > config file defaults > SDK defaults
