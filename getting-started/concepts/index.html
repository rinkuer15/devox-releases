Core Concepts

Devox orchestrates AI coding agents through several core concepts. Understanding these will make everything else click.

Projects (Codebases)

Section titled “Projects (Codebases)”

A project is a git repository that Devox manages. Register one via the Web UI (paste a GitHub URL or local path). Devox clones it to ~/.devox/workspaces/owner/repo/source/ and uses it as the base for all workflow runs.

Workflows

Section titled “Workflows”

A workflow is a YAML file that defines a multi-step AI coding task as a directed acyclic graph (DAG). Each workflow lives in .devox/workflows/ and has a name, description, and a set of nodes with declared dependencies.

Nodes without dependencies run immediately. Nodes in the same dependency layer run in parallel. This means a workflow with three independent review nodes will fan out and run all three concurrently, then converge at a downstream node that depends on all of them.

Devox ships with bundled default workflows. Run devox workflow list to see what’s available, or browse .devox/workflows/defaults/ for real examples.

Nodes

Section titled “Nodes”

Nodes are the building blocks of workflows. Each node does exactly one thing, and every node must specify exactly one of six types:

TypeWhat it does
command:Loads a command file from .devox/commands/ and sends it to an AI agent
prompt:Sends an inline prompt string to an AI agent
bash:Runs a shell script (no AI). Stdout is captured as $nodeId.output
loop:Runs an AI prompt repeatedly until a completion signal is detected
approval:Pauses the workflow for human review (approve or reject)
cancel:Terminates the workflow early with a reason string

Nodes connect through depends_on to form a DAG. You can add conditional branching with when: expressions, control join behavior with trigger_rule, and override the AI provider or model per node.

Commands

Section titled “Commands”

A command is a markdown file in .devox/commands/ that serves as an AI prompt template. When a workflow node references command: investigate-issue, Devox loads .devox/commands/investigate-issue.md, substitutes variables, and sends the result to the AI.

Commands support variable substitution. The most commonly used variables:

VariableResolves to
$ARGUMENTSThe user’s input message
$ARTIFACTS_DIRPre-created directory for workflow artifacts
$BASE_BRANCHThe base branch (auto-detected or configured)
$DOCS_DIRDocumentation directory path (default: docs/)
$WORKFLOW_IDUnique ID for the current workflow run

See the Variable Reference for the complete list.

Devox ships with bundled default commands for common operations like investigation, implementation, and code review. Repo-level commands in .devox/commands/ override bundled defaults with the same name.

Isolation (Worktrees)

Section titled “Isolation (Worktrees)”

Every workflow run gets its own git worktree by default — an isolated copy of your repository. This gives you three things:

  1. Your working branch stays clean. Workflow changes happen in a separate directory.
  2. Multiple workflows run in parallel without conflicting with each other.
  3. Failed runs don’t leave a mess. Clean up with devox isolation cleanup.

Worktrees live at ~/.devox/workspaces/<owner>/<repo>/worktrees/. Each worktree gets its own branch, so you can inspect the work, create a PR from it, or discard it.

Providers

Section titled “Providers”

A provider is an AI engine integration. Devox ships three built-in providers:

ProviderIDBinary Required
Claude Code (Anthropic)claudeYes — ~/.local/bin/claude
Codex CLI (OpenAI)codexYes — ~/.devox/vendor/codex/codex
GitHub Copilot CLIcopilotYes — gh copilot or @github/copilot-cli

See AI Assistants for setup instructions.

Skills

Section titled “Skills”

Skills are reusable SKILL.md files that inject domain knowledge into Claude nodes. They are stored in .claude/skills/{name}/SKILL.md and referenced by name in workflow nodes via skills: [name]. Devox provides a Skills page in the Web UI to install and manage them.
